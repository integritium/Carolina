# 4. Proposed Framework

Bagian ini menguraikan desain framework PRNG yang menggabungkan 2D Linear Cellular Automata sebagai lingkungan dan agen-agen MCTS sebagai sumber entropi tambahan melalui gamifikasi.

## 4.1 Desain Lingkungan: 2D LCA sebagai Arena Permainan

Framework yang diusulkan menggunakan grid [[2D LCA]] berdimensi $N \times N$ sebagai "arena permainan." Setiap sel pada grid berada dalam salah satu dari dua state: hidup (1) atau mati (0). Pada setiap langkah waktu, aturan transisi LCA linear diterapkan secara simultan ke seluruh sel menggunakan [[Von Neumann neighborhood]], menghasilkan konfigurasi baru. Dalam framework konvensional (tanpa agen), proses ini berjalan secara otonom — state berikutnya sepenuhnya ditentukan oleh state saat ini dan aturan transisi. Framework yang diusulkan menambahkan lapisan kompleksitas baru: setelah setiap langkah evolusi CA, satu atau lebih agen mendapat giliran untuk *mengintervensi* grid dengan mengubah (*flipping*) state sejumlah sel tertentu.

Lingkungan beroperasi dalam siklus berulang dengan struktur berikut: pertama, aturan CA diterapkan untuk mengevolusi grid. Kedua, agen (jika ada) memilih sel mana yang akan di-flip berdasarkan strategi MCTS-nya. Ketiga, state grid yang telah dimodifikasi menjadi input untuk langkah evolusi berikutnya. Siklus ini berulang hingga jumlah bit yang diinginkan telah terkumpul. Struktur ini dapat diformalisasikan sebagai:

$$G^{t+1} = \text{Agent}\Big(\varphi(G^t)\Big) \tag{5}$$

di mana $G^t$ adalah state grid pada waktu $t$, $\varphi$ adalah fungsi transisi CA, dan $\text{Agent}(\cdot)$ merepresentasikan intervensi agen terhadap state grid pasca-evolusi. Dalam konfigurasi *environment-only* (tanpa agen), $\text{Agent}(\cdot)$ adalah fungsi identitas.

> [!important] Tiga Konfigurasi Agen
> Framework dievaluasi dalam tiga konfigurasi:
> 1. **Environment-only (0 agen):** CA berevolusi secara otonom tanpa intervensi. Ini adalah baseline.
> 2. **1 MCTS Agent + Environment:** Satu agen cerdas berinteraksi dengan lingkungan CA. Lingkungan bertindak sebagai "lawan pasif" yang mengikuti aturan transisi tetap.
> 3. **2 MCTS Agents + Environment:** Dua agen berkompetisi — satu *maximizer* (memaksimalkan sel hidup) dan satu *minimizer* (meminimalkan sel hidup). Konflik tujuan ini menghasilkan dinamika yang paling tidak terduga.

## 4.2 Mekanisme Generasi Bilangan Acak

Proses generasi bilangan acak dari grid CA menggunakan mekanisme yang disebut *file append* dengan kedalaman $d$. Pada setiap siklus, sebanyak $d$ baris terakhir dari konfigurasi grid diekstrak dan ditambahkan (*appended*) ke file output biner. Konfigurasi ini menghasilkan $d \times N$ bit per siklus. Dengan total target 125 juta bit (sesuai rekomendasi NIST untuk evaluasi statistik), jumlah siklus yang diperlukan bervariasi tergantung pada ukuran grid dan kedalaman append:

$$\text{Jumlah siklus} = \left\lceil \frac{125{,}000{,}000}{d \times N} \right\rceil \tag{6}$$

Mekanisme *file append depth* $d$ memiliki implikasi penting: kedalaman yang terlalu kecil memerlukan lebih banyak siklus (dan dengan demikian lebih banyak evolusi CA), sementara kedalaman yang terlalu besar berisiko menangkap korelasi spasial dalam grid CA yang belum sepenuhnya "tercampur." Jurnal ini menggunakan $d = 64$ sebagai nilai default berdasarkan studi Kılıç (2024), dengan catatan bahwa Random Excursion dan Random Excursion Variant Tests adalah tes yang paling sensitif terhadap nilai $d$ dan paling sering gagal.

> [!info] Definisi File Append Depth
> **File append depth** ($d$): Jumlah baris grid CA 2D yang diekstrak pada setiap siklus untuk ditambahkan ke file output biner. Nilai $d$ harus dipilih secara hati-hati — terlalu kecil meningkatkan waktu generasi, terlalu besar mengurangi kualitas keacakan karena korelasi spasial.

## 4.3 Peran dan Strategi Agen

Setiap agen dalam framework memiliki dua komponen utama: *goal* (tujuan) dan *mechanism* (mekanisme pencarian). Agen *maximizer* bertujuan memaksimalkan jumlah sel hidup di grid, sedangkan agen *minimizer* bertujuan meminimalkannya. Kedua agen menggunakan [[MCTS]] dengan kebijakan [[Negamax]] untuk menentukan aksi terbaiknya.

### 4.3.1 Representasi Aksi

Setiap aksi agen terdiri dari pemilihan tepat $K$ sel pada grid untuk di-*flip* (mengubah state dari 0 ke 1 atau sebaliknya). Parameter $K$ (disebut *K Value* atau *control actions count*) menentukan "kekuatan" intervensi agen pada setiap giliran. Jurnal ini mengevaluasi tiga nilai: $K = 10$, $K = 20$, dan $K = 40$. Semakin besar $K$, semakin besar pengaruh agen terhadap evolusi grid — tetapi juga semakin besar ruang pencarian yang harus dijelajahi MCTS:

$$|\text{Ruang aksi}| = \binom{N^2}{K} \tag{7}$$

Untuk grid $20 \times 20$ dan $K = 40$, ruang aksi adalah $\binom{400}{40}$ — angka astronomis yang membuat pencarian exhaustive mustahil. Inilah mengapa MCTS diperlukan: algoritma ini menjelajahi hanya sebagian kecil dari ruang pencarian namun secara statistik konvergen menuju aksi yang mendekati optimal.

### 4.3.2 Parameter MCTS

MCTS dalam framework ini dikonfigurasi dengan parameter-parameter berikut: jumlah iterasi MCTS per keputusan tidak dilaporkan secara eksplisit, tetapi *child amount* ditetapkan sebagai $C = 3$ (setiap node memiliki maksimal 3 anak). Konstanta eksplorasi UCB1 menggunakan nilai default $\sqrt{2}$. Dalam konfigurasi 2 agen, urutan giliran bersifat tetap: {agen, lingkungan, agen, lingkungan, ...}, sehingga setiap agen secara bergantian berinteraksi dengan grid pasca-evolusi CA.

```
Algorithm 1: MCTS Agent Decision
Input:  Grid state G^t (N×N binary grid), K (jumlah sel yang di-flip), 
        player_type ∈ {maximizer, minimizer}
Output: Action a* (set K sel yang akan di-flip)

1:  root ← CreateNode(G^t)
2:  for iteration = 1 to MAX_ITER do
3:      node ← TreePolicy(root)          // Selection + Expansion (UCB1)
4:      Δ ← NegamaxEval(node, depth)     // Simulation via Negamax
5:      Backpropagate(node, Δ)            // Update statistics
6:  end for
7:  a* ← BestChild(root)                 // Most visited child
8:  Apply a* to G^t → G^t (modified)
9:  return G^t (modified)

Total: Depends on MAX_ITER and search depth; significant overhead vs. pure CA evolution
```

Setelah agen memutuskan aksinya, state grid dimodifikasi in-place, dan evolusi CA dilanjutkan. Proses ini berulang dalam loop utama generasi bilangan acak.

## 4.4 Alur Keseluruhan Framework

Alur kerja lengkap framework dapat dirangkum sebagai berikut:

1. **Inisialisasi:** Grid $N \times N$ diinisialisasi menggunakan [[Mersenne Twister]] dengan densitas yang sesuai dengan aturan CA yang dipilih.
2. **Loop Generasi:** Pada setiap siklus:
   - a. Terapkan aturan transisi CA ke seluruh grid → $G' = \varphi(G^t)$
   - b. Jika ada agen: agen mengambil keputusan via MCTS dan memodifikasi $G'$
   - c. Jika ada 2 agen: agen kedua mengambil keputusan dan memodifikasi $G'$ lebih lanjut
   - d. Ekstrak $d$ baris terakhir dari $G'$ dan append ke file output
   - e. $G^{t+1} \leftarrow G'$
3. **Terminasi:** Loop berhenti setelah 125 juta bit terkumpul.
4. **Evaluasi:** File output diuji menggunakan NIST Statistical Test Suite.

> [!check] Reprodusibilitas
> - **Kode:** Tidak disebutkan secara eksplisit apakah kode tersedia publik
> - **Dataset:** 210 file biner yang dihasilkan, masing-masing 125 juta bit (~25.6 GB total). Tidak ada URL repositori
> - **Hardware:** Informasi ini tidak dijelaskan dalam jurnal (tidak disebutkan spesifikasi GPU/CPU/RAM)
> - **Random seed:** Menggunakan Mersenne Twister dari C++ standard library untuk inisialisasi, seed spesifik tidak dilaporkan
> - **Hyperparameter:** $K \in \{10, 20, 40\}$, grid $\in \{10\times10, 15\times15, 20\times20\}$, child amount $C = 3$, 7 aturan LCA, file append depth $d = 64$
> - **Catatan:** Tanpa kode sumber dan seed yang dilaporkan, reproduksi eksak akan sulit. Namun, reproduksi statistik (tren umum hasil) seharusnya memungkinkan dengan implementasi ulang.

## 4.5 Fungsi Transisi State dan Interaksi Lingkungan

Dalam konfigurasi multi-agen, interaksi antara agen dan lingkungan mengikuti protokol terstruktur. Lingkungan (*environment*) bukan merupakan agen pasif — ia menerapkan aturan transisi CA yang secara aktif mengubah state grid. Ini berarti setiap agen harus memperhitungkan bukan hanya aksi lawan (agen lain), tetapi juga "aksi" lingkungan (evolusi CA). Inilah yang memotivasi penggunaan [[Negamax]] sebagai kebijakan backup: karena lingkungan "bermain" menurut aturan tetap, agen harus memperlakukannya sebagai lawan yang perilakunya harus diantisipasi.

Jurnal ini mendefinisikan urutan giliran (*turn order*) sebagai **{agent, environment, agent, environment, ...}**. Dalam konfigurasi 2 agen, urutannya menjadi **{agent 1, environment, agent 2, environment, ...}** — masing-masing agen dan lingkungan bergantian secara round-robin. Perlu dicatat bahwa urutan giliran yang berbeda dapat menghasilkan dinamika yang berbeda; jurnal ini tidak mengeksplorasi variasi urutan giliran, yang merupakan potensi arah penelitian masa depan.

> [!question] Bagian Tidak Jelas
> Jurnal tidak secara eksplisit menjelaskan bagaimana *interaction sequence* (urutan giliran) ditangani dalam konfigurasi 2 agen. Apakah kedua agen mengambil keputusan berdasarkan grid yang sama (sebelum evolusi CA), atau agen kedua melihat grid yang sudah dimodifikasi oleh agen pertama? Berdasarkan konteks, interpretasi yang paling masuk akal adalah yang kedua — agen kedua melihat state pasca-modifikasi agen pertama, yang kemudian diikuti evolusi CA.
